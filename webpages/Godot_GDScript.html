GDScript Language & Godot Engine




× Introduction Pros and Cons Conclusion

Introduction:

GDScript is the primary language used for programming games in the Godot engine. While the engine supports C#, GDScript is designed to use fewer and simpler lines of code to accomplish the same tasks. Since it was made for the engine, it is the language with the most support. Game developers experienced in C# may want to use that instead for the sake of not having to learn GDScript, but GDScript is simple to learn and any programmer that knows Python is likely to be able to jump into GDScript right away, as it was designed to have similar syntax to Python. C# is faster with raw calculations, and GDScript is faster at accessing Godot’s internal engine functions.



Pros and Cons:

Pros:

  • GDScript is simple to learn and use:
    • It is a sufficient language for much of development done on Godot, C# will take more time to learn.
  • The learning curve is easy:
    • Learning to use the engine and the technology is less complicated than other popular engines.
  • Includes many features:
    • Many features a game developer would want for creating a game, even to professional standards.
  • Godot is completely free:
    • It has no licensing that requires royalties or any sort of fees.
  • Godot is a lightweight engine:
    • The download for the engine is very small and quick, and can be run on less powerful machines.
  • Supports 2D and 3D games:
    • Support for the 2D engine is the strongest, but the 3D engine is still powerful.
  • Godot is open source and can be fully modified to fit the needs of a developer.
  • Projects can be easily ported across all major platforms.

Cons:

  • The engine is behind technologically:
    • Compared to other engines, a large enough professional studio would find it lacking for high budget titles.
  • 3D support is lacking:
    • Its 3D rendering, animations, and physics are not up to par with other engines.
  • Godot has a small asset library:
    • It has fewer contributions compared to the asset stores of other engines.
  • C# has room for improvement:
    • Using strictly C# means certain features not being available or poor performance, and the Code doesn't recompile until the project is run.


Conclusion:

Godot is a solid platform for any developer or team wanting to create a 2D game with ease and simplicity. For the 3D engine, the use cases are limited to smaller teams and independent developers wanting to create a lower budget 3D game. It is insufficient for larger, professional teams that expect a game that has high-quality graphics while maintaining acceptable performance. It would not be suitable for competitive games that require quick reactions and fluid feedback to player input.

It is a good alternative to Roblox for learning game development, because it follows the standard design of a game engine compared to Roblox. It has no pre-existing style or playerbase, so the games you make are going to be that much more unique. Godot projects can be published on any game store and isn’t bound to a specific platform unlike Roblox. Projects start essentially from scratch meaning the developers need to be able cover all logic and movement. It provides a lot of functions and Nodes to assist with development. There are trade-offs, and Roblox still has its merits for certain use cases. If a developer has fewer experience with certain aspects of development, Godot will be more difficult to create a finished game with, but for a team that has developers with all the needed skills, Godot is a better choice for building skills and projects that are more relevant across the industry.

Demo #1 - Basic Top-Down 2D Player Movement:

Starting with a blank Godot project, I set up the two scenes that will be needed for this demo. The first scene I called "main". This will be the "playing" area of the project that the Player will move around in. The second scene I called "player". This scene will be used for everything needed for the player. The Node I selected for the main Node of the player scene is a "CharacterBody2D", which has a "CollisionShape2D" Node underneath it to define collision, and a "Sprite2D" Node which is where we can put the Player's actual visible texture. For the sake of simplicity for this demo, I'm using the default "icon.svg" file as a placeholder for a real player sprite. Since this "sprite" has a square shape, I have set the CollisionShape2D a square shape as well, which covers the player sprite. Collision won't be used in this demo, but the editor will label the CharacterBody2D Node with a warning, so it might as well be included.



The next step is to create movement inputs in the InputMap section of the Project settings. I've named them "move_left", "move_right", "move_up", and "move_down", then I mapped each one to their respective "WASD" inputs. You could also choose to use the arrow keys instead, or add them as additional inputs and use both schemes.



Now for coding the Player movement itself. I attached a new script to the CharacterBody2D Node, and included the following line at the top of the script: 

extends CharacterBody2D

Next, I initialized some variables that will be used in the script. 

var screen_size
## The keyword '@export' is used so that the variable can be changed in the editor without touching any code.
@export var speed = 400

The following function is called when the Node and its children enter the scene. Here it's used to get the size of the player's screen.

func _ready():
	screen_size = get_viewport_rect().size

Lastly, the following function will be called each frame prior to rendering. The actual movement code is handled here.

func _process(delta):
	## Set the Player velocity to (0.0, 0.0)
	velocity = Vector2.ZERO		
	## Get a normalized vector of the four movement Inputs
	var direction = Input.get_vector("move_right","move_left","move_up","move_down")
	## Set player velocity to the given speed in the direction of the Input
	velocity = direction * speed
	## Get Player position, and clamp the Position to minimum vector coordinates
	position += velocity * delta
	position = position.clamp(Vector2.ZERO, screen_size)

Now, to test if everything works. In order for the player to function, I had to initialize the player scene itself as a child of the main Node of the main scene from earlier.





Demo #2 - Basic Third-Person 3D Player Movement:

This demo is similar to the previous one, but in 3D. As such, setting up the Main scene is a lot more involved and there are more complicates. Firstly, since the 3D space is an open void, the "backdrop" does not act as a suitable ground for the Player to move around on. Secondly, a Camera Node that is set up to view the Player as well as follow along with their movement is needed. Lastly, the movement and inputs are essentially the same, but the X and Z axis need to be accounted for horizontal movement seperately from vertical movement which takes place in the form of a jump action, which requires an addtional input which I called "jump", and I mapped it to Spacebar, as well as gravity which is constant and passive.


Starting with the Main Scene, I set the main Node as a generic Node. Underneath I created a StaticBody3D Node which will be used as the "Ground" object. Of course, this Node alone is not enough for a physical ground. It needs a CollisionShape3D Node to create the collision box, and I set its size to (60.0,2.0,60.0). It also needs a MeshIntance3D with a BoxMesh for the visible aspect of the Ground, set to the same size as the collision box. Another Node that needs to be underneath the main Node is a DirectionalLight3D Node so that the scene isn't dark. I Manipulated the rotation of the light until it illuminated the Ground.


Now for the Player scene. For the main Node, I selected CharacterBody3D, and underneath that a Node3D to hold the character model. Unlike 2D, Godot doesn't provide a default 3D model in the project folder, so I downloaded assets from the tutorial within their documentation and downloaded Blender so that I could import that model into my project and place it within the Node3D under the Player Node. Finally, I added a CollisionShape3D with the Sphere shape to create a roughly accurate collision bubble around the Player. I also made sure that the bottom of the Collision lies along the "floor" grid to prevent the player from slightly hovering above or partially clipping under the ground.


The next part is scripting the Player movement itself:

extends CharacterBody3D


## Initialize some variables that are once again exported
@export var speed = 14
@export var fall_acceleration = 75
@export var jump_impulse = 20	


## This variable will for calculating the velocity that the player should be attemtping to reach
var target_velocity = Vector3.ZERO

Next, the _physics_property() function which works similar to _process() but is better suited for handling physics.

func _physics_process(delta):
	## This vector is in 2D, technically (X,Y) but Y will be used as Z in the 3D Vector
	var direction = Input.get_vector("move_left","move_right","move_forward","move_back")
	## Convert to a 3D vector to map the horizontal movement inputs with the X and Z directions, Y=0
	direction = Vector3(direction.x, 0 , direction.y)
	
	## Handle the "jump" action and manipulate vertical velocity respectively
	if Input.is_action_pressed("jump") and is_on_floor():
		target_velocity.y = jump_impulse
	
	## Control the 3D direction that the Player model is facing
	if direction != Vector3.ZERO:
		$Pivot.basis = Basis.looking_at(direction)
		
		## Set the velocity as a set speed towards the correct directions
		target_velocity.x = direction.x * speed
		target_velocity.z = direction.z * speed
		 
		## Calculate constant, passive gravity as long as the player is in the air
		if not is_on_floor():
			target_velocity.y = target_velocity.y - (fall_acceleration * delta)
		
		## Move the player
		velocity = target_velocity
		move_and_slide()

This code has similarities to the previous Demo, but needs to take an additional dimension into account as well as gravity and a "jump" action.


Now that player movement is coded, there is still one more major part of this demo. Before that, the Player scene needs to be initialized as a Node within the main scene.

I added a Marker3D Node underneath the Player Node. This Node will serve the purpose of rotating the Camera to find the best angle. Underneath that, I created a Camera3D Node which needs to be moved to a suitable place in combination with the Marker3D's rotation to get a preffered view of the Player. Since the Camera is not scripted, it won't rotate or move with the player's movement to maintain this exact view, having the Camera as a child of the Player Node in the main scene will at least allow it to move as such that the Camera's distance from the Player will stay the same.


With the main and player scenes set up, the movement coded, and the Third-Person Camera in place, everything can be playtested together.
